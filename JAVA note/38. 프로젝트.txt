2. 기획 산출물(UI 관련)
    1) 와이어프레임 (Wireframe) : UI 중심의 화면 레이아웃
    2) 목업 
    3) 프로토타입
    4) 화면설계서 (SB)
        1] 정책, 프로세스, 와이어프레임, 디스크립션 등이 모두 포함된 설계 문서     
        2] 웹이 대부분 화면을 기반으로 되어 있어서 나온 용어
        3] 화면을 구성하기 위해 기획과정에서 나온 모든 것을 포함함. 
            - 기획 작업의 총 집합 문서
            - 웹 기획의 최종 산출물
                - 화면 UI 설계 + 기능정의서 + 업무흐름도
                - 시스템과의 인터페이스에서 발생하는 다양한 사용자케이스를 감안하여
                  전달하는 정보나 인터렉션 정의까지 포함하여 작성.
        4] 이용 시나리오와 인터렉션을 강조하여 전반적인 흐름을 보여주는 문서.

3. 화면설계서의 구성요소
    1) 수정 이력 관리
    2) 기획의도
    3) 프로세스 플로우차트
    4) 개발 범위 화면 목록(IA)
    5) 와이어프레임
    6) 디스크립션 - 노출 로직, 디자인, 솔루션 사용 등 설명

4. 화면설계서(스토리보드,SB0 디스크립션의 작성)
    1) 이를 닦는것
        - ①칫솔을 든다
          ②치약을 든다
          ③치약을 짠다
          ④칫솔을 위아래로 움직인다

        - ①칫솔을 든다
          ②치약을 든다
          ③칫솔에 치약을 짠다
          ④칫솔을 이빨에 댄다
          ⑤칫솔을 위아래로 움직인다.

        - ①칫솔을 든다
          ②마음에 드는 치약을 고른다
          ③고른 치약을 칫솔에 짠다
          ④칫솔을 이빨에 댄다
          ⑤칫솔을 위아래로 움직인다.
          ⑥칫솔을 좌우로 움직인다.
          ⑦칫솔을 안팎으로 움직인다.
          ⑧칫솔로 혀를 닦는다.
          ⑨물컵을 든다
          ⑩물컵에 물을 받는다
          ⑪입을 헹군다
    2) 디스크립션은 세부적으로 쪼개서 명확하고 정확하게 작성해야함.
    3) 인터랙션과 데이터 노출에 대한 부분의 로직을 분명히 해야함.
    4) 개발과 디자인시안에 관련되어 이슈사항이나 우회사항이 있다면 명시해줘야 함.
    5) 정책이 불분명한 부분은 명시하여 추후 버전관리가 되도록 해야 함.

5. 사용자의 흐름과 데이터의 흐름
    1) 함께 파악해야함.
    2) 사용자의 흐름
        - Flow Chart
        - 순서도
        - Story Board
            - 회원가입, 구매하기, 환불/반송, 온라인문의 등 주요 서비스의 플로우차트를 통해
              user case를 파악하고 사용자의 흐름을 설계.
    3) 데이터 흐름 
        - ERD (Entity Relationship Diagram)
        - 데이터의 흐름은 프로그래머에 의해서 데이터베이스 설계로 만들어짐.
        - DBA(DataBase Administer)가 수행하는 경우도 있음.

7. 요구사항
    1) 기능적 요구사항
        - 시스템의 사용자가 직접 이용하는 개별적인 기능
        - 특정 입력에 대해 시스템이 산출하는 출력에 대한 처리 과정
    2) 비기능적 요구사항
        - 성능, 신뢰성, 보안성, 안정성, 
        - 제품의 품질 기준 만족시키기 위해 소프트웨어가 가져야하는 사항
        - 시스템 기능에 관련되지 않은 제약사항
